AI Agents in Depth

Bojie Li · Design Principles and Engineering Practice · 2026

Книга рассматривает агента как систему из модели, контекста и инструментов, чьи возможности определяются не только весами LLM, но и инженерной обвязкой. Контекст, память и знания задают наблюдаемую картину мира; инструменты и код превращают решения в проверяемые действия; оценивание и постобучение связывают поведение с измеримым улучшением. Самоэволюция, мультимодальное взаимодействие и команды агентов расширяют эту основу, одновременно создавая новые риски задержки, доступа, каскадных ошибок и накопления плохого опыта. Главный практический вывод — проектировать наблюдение, действие и проверку как единый воспроизводимый цикл, в котором модель и Harness развиваются вместе.The book treats an agent as a system composed of a model, context, and tools, whose capabilities are determined not only by the LLM's weights but also by its engineering harness. Context, memory, and knowledge define the observable view of the world; tools and code turn decisions into verifiable actions; evaluation and post-training connect behavior to measurable improvement. Self-evolution, multimodal interaction, and teams of agents extend this foundation while introducing new risks involving latency, access, cascading errors, and the accumulation of poor experience. The main practical conclusion is to design observation, action, and verification as a single reproducible loop in which the model and harness evolve together.

1Основы агентов и harness-инжинирингAgent Foundations and Harness Engineering

Раздел задаёт минимальную архитектуру агента: языковая модель принимает решения, контекст определяет доступную ей картину мира, а инструменты превращают намерение в действие. Инженерная обвязка (Harness) ограничивает, проверяет и исправляет этот цикл, поэтому автономность рассматривается вместе с наблюдаемостью и безопасностью.This section establishes the minimal architecture of an agent: the language model makes decisions, context determines the view of the world available to it, and tools turn intent into action. The engineering harness constrains, verifies, and corrects this loop, so autonomy is considered together with observability and safety.

1Agent = LLM + Context + Tools

LLM выполняет понимание, планирование и выбор следующего действия; контекст содержит всё, что модель видит в момент решения; инструменты определяют всё, что система может сделать во внешнем мире. Слабость любого из трёх элементов ограничивает всю систему: интеллект не компенсирует отсутствующую информацию или невозможность действовать. Автор даёт три эквивалентных языка описания: реализационный — LLM, Context, Tools; интуитивный — мозг, глаза, руки и ноги; академический — policy, observation space, action space. В таблице продуктов Coding Agent видит репозиторий и терминал, Search Agent — веб и базы, а Pine AI — данные пользователя и сервисные знания; все они планируют, действуют и меняют стратегию по обратной связи. Современная релевантность: При проектировании агента полезно отдельно специфицировать матрицу наблюдений и действий: какие инструкции, состояние, память и результаты доступны модели; какие инструменты разрешены; какие данные устаревают; где нужны подтверждения.The LLM handles understanding, planning, and selection of the next action; context contains everything the model can see at the moment of decision; tools determine everything the system can do in the external world. A weakness in any of the three elements limits the entire system: intelligence cannot compensate for missing information or an inability to act. The author offers three equivalent descriptive vocabularies: implementation-oriented—LLM, Context, Tools; intuitive—brain, eyes, hands and feet; and academic—policy, observation space, action space. In the product comparison table, a Coding Agent sees a repository and terminal, a Search Agent sees the web and databases, and Pine AI sees user data and service knowledge; all of them plan, act, and adjust their strategy based on feedback. Modern relevance: when designing an agent, it is useful to specify the observation and action matrix separately—which instructions, state, memory, and results are available to the model; which tools are permitted; which data becomes stale; and where confirmation is required.

LLM — мозг, контекст — глаза, инструменты — руки и ноги: хорошее решение возникает только при совместной работе восприятия, мышления и действия.The LLM is the brain, context is the eyes, and tools are the hands and feet: a sound decision emerges only when perception, reasoning, and action work together.

Agent = Brain + Eyes + Hands and Feet.

2ReAct Loop and Agent Trajectory

ReAct чередует reasoning, acting и observation: модель выбирает действие, framework исполняет инструмент, результат добавляется в историю, после чего модель принимает следующее решение. Контекст каждого вызова равен статическому префиксу — system prompt и tool definitions — плюс растущая trajectory из пользовательских, ассистентских и инструментальных сообщений. В Experiment 1-1 агент агрегирует выручку в разных валютах: сначала параллельно получает курсы, затем вычисляет сумму кодом, затем выдаёт ответ. Абляция показывает причинные роли компонентов: без tool definitions нет действия, без tool results возникает слепое повторение, без reasoning решения конфликтуют, без history агент заново проходит уже завершённые шаги. Современная релевантность: Книга показывает, что траектория делает работу агента интерпретируемой, позволяет анализировать паттерны, улучшать инструменты и превращать опыт в базу знаний или данные для RL.ReAct alternates reasoning, acting, and observation: the model selects an action, the framework executes the tool, the result is added to the history, and the model then makes its next decision. The context of each call consists of a static prefix—the system prompt and tool definitions—plus a growing trajectory of user, assistant, and tool messages. In Experiment 1-1, the agent aggregates revenue in different currencies: it first retrieves exchange rates in parallel, then computes the total in code, and finally returns the answer. The ablation reveals the causal role of each component: without tool definitions there is no action; without tool results, blind repetition occurs; without reasoning, decisions conflict; without history, the agent repeats steps it has already completed. Modern relevance: the book shows that a trajectory makes an agent's work interpretable, supports pattern analysis and tool improvement, and turns experience into a knowledge base or data for reinforcement learning.

Траектория похожа на судовой журнал: она не управляет кораблём сама, но сохраняет решения, наблюдения и пройденный путь, без которых следующий манёвр становится слепым.A trajectory is like a ship's log: it does not steer the vessel itself, but it preserves decisions, observations, and the route already traveled, without which the next maneuver becomes blind.

This “think → do → see → think → do → see” loop repeats until the task is done.

3Model + Harness

Harness — весь исполняющий и защитный слой вне модели. Его ядро составляют Context и Tools, а производственную надёжность обеспечивают Constrain, Verify и Correct: ограничить допустимое, проверить фактический результат, восстановиться, откатиться или передать задачу человеку после сбоя. В примере возврата политика семи дней относится к Context, API `query_order` и `process_refund` — к Tools, лимит суммы — к Constrain, сверка с базой — к Verify, повтор после тайм-аута — к Correct. Одна модель без обвязки может заявить о несуществующем возврате; с ней проходит проверяемый путь исполнения. Современная релевантность: Современный harness включает управление состоянием, разрешения по принципу fail-safe defaults, валидацию структурированных результатов, circuit breaker, retries с лимитами, откат, передачу человеку и наблюдаемость. Его следует версионировать и оценивать отдельно от модели, поскольку изменение обвязки способно изменить качество системы при неизменных весах.The harness is the entire execution and protection layer outside the model. Its core consists of Context and Tools, while production reliability is provided by Constrain, Verify, and Correct: limit what is permitted, verify the actual outcome, and recover, roll back, or hand the task off to a human after a failure. In the refund example, the seven-day policy belongs to Context; the `query_order` and `process_refund` APIs are Tools; the amount limit is Constrain; reconciliation against the database is Verify; and retrying after a timeout is Correct. A model alone may claim that a nonexistent refund has been issued; with a harness, it follows a verifiable execution path. Modern relevance: a modern harness includes state management, fail-safe-default permissions, validation of structured outputs, a circuit breaker, bounded retries, rollback, human handoff, and observability. It should be versioned and evaluated separately from the model because changing the harness can change system quality even when model weights remain unchanged.

Как упряжь направляет силу лошади, а системы безопасности гоночного автомобиля сдерживают последствия ошибки пилота, harness превращает мощность модели в управляемое выполнение.Just as a harness directs a horse's strength, while a race car's safety systems contain the consequences of driver error, an agent harness turns model capability into controlled execution.

Agent = LLM + [Context + Tools + Constrain + Verify + Correct] = Model + Harness

4Workflow vs. Autonomous Agent

Workflow фиксирует граф шагов кодом и отдаёт LLM только локальные задачи внутри узлов. Автономный агент строит путь во время выполнения по обратной связи. Первый обеспечивает предсказуемость и соблюдение инвариантов, второй адаптируется к открытым задачам, но увеличивает стоимость, задержку и вероятность накопления ошибок. В бронировании перелёта workflow жёстко упорядочивает идентификацию, поиск, оплату и подтверждение. Автономный вариант может обнаружить необходимость входа, вернуться к поиску и уточнить отношение пользователя к пересадкам. Книга рекомендует лестницу: один LLM-вызов, затем workflow, затем автономия; критичные и исследовательские участки можно смешивать. Современная релевантность: Критичные процессы со строгими требованиями соответствия разумно фиксировать как workflow с явными инвариантами. Открытые исследовательские, coding- и computer-use-задачи подходят автономному контуру. Для него нужны max iterations, условия завершения, бюджет, песочница, журналирование и human-in-the-loop checkpoints.A workflow fixes the graph of steps in code and delegates only local tasks within its nodes to the LLM. An autonomous agent constructs its path at runtime based on feedback. The former provides predictability and preserves invariants; the latter adapts to open-ended tasks but increases cost, latency, and the likelihood of accumulating errors. In flight booking, a workflow rigidly orders identification, search, payment, and confirmation. An autonomous version may discover that authentication is required, return to search, and clarify the user's tolerance for layovers. The book recommends a progression: one LLM call, then a workflow, then autonomy; critical and exploratory segments can be combined. Modern relevance: critical processes with strict compliance requirements are best fixed as workflows with explicit invariants. Open-ended research, coding, and computer-use tasks suit an autonomous loop. Such a loop requires maximum iteration limits, termination conditions, a budget, a sandbox, logging, and human-in-the-loop checkpoints.

Workflow — железнодорожный маршрут с заранее уложенными путями; автономный агент — водитель, который выбирает дорогу по текущей обстановке, но обязан соблюдать границы и уметь остановиться.A workflow is a railway route with tracks laid in advance; an autonomous agent is a driver who chooses the road based on current conditions but must respect boundaries and be able to stop.

When building an LLM application, go from simple to complex.

5Defense-in-Depth Guardrails

Guardrails реализуют функции Constrain, Verify и Correct несколькими независимыми слоями. Входной слой фильтрует нерелевантные и вредоносные инструкции, исполнительный оценивает риск вызовов инструментов, выходной проверяет ответ и утечки данных; человеческая эскалация закрывает ситуации высокого риска и исчерпания попыток. Книга различает jailbreak — прямую попытку пользователя обойти ограничения — и prompt injection через внешние данные. Риск инструмента зависит от обратимости, разрешений и финансового эффекта. Отмена заказа, крупный возврат и платёж названы примерами действий, которые должны требовать подтверждения, особенно на ранних стадиях внедрения. Современная релевантность: Подтверждённое книгой ядро политики безопасности — fail-safe defaults, минимальные права, оценка риска по обратимости, разрешениям и финансовому эффекту, проверка структурированных полей и отдельное подтверждение операций высокого риска. Эти барьеры должны оставаться независимыми, чтобы отказ одного слоя не отменял остальные.Guardrails implement the Constrain, Verify, and Correct functions through several independent layers. The input layer filters irrelevant and malicious instructions, the execution layer assesses the risk of tool calls, and the output layer checks responses and data leakage; human escalation covers high-risk situations and exhausted retry limits. The book distinguishes a jailbreak—a direct attempt by a user to bypass restrictions—from prompt injection delivered through external data. Tool risk depends on reversibility, permissions, and financial impact. Canceling an order, issuing a large refund, and making a payment are cited as actions that should require confirmation, especially during early deployment. Modern relevance: the safety policy core supported by the book consists of fail-safe defaults, least privilege, risk assessment based on reversibility, permissions, and financial impact, validation of structured fields, and separate confirmation for high-risk operations. These barriers must remain independent so that the failure of one layer does not disable the others.

Эшелонированная защита похожа на авиационную безопасность: отдельная проверка может ошибиться, поэтому независимые барьеры последовательно предотвращают, обнаруживают и локализуют отказ.Defense in depth resembles aviation safety: any individual check may fail, so independent barriers successively prevent, detect, and contain a failure.

Think of guardrails as defense in depth.

2Инженерия контекстаContext Engineering

Контекст — не стенограмма, а управляемое состояние вычисления: порядок сообщений, стабильный префикс, инструкции, навыки и результаты инструментов напрямую влияют на решения модели. Раздел показывает, как удерживать информационную плотность, использовать кеширование и изолировать объёмную работу, не теряя проверяемых фактов.Context is not a transcript but managed computational state: message order, a stable prefix, instructions, skills, and tool results directly shape the model’s decisions. This section explains how to preserve information density, use caching effectively, and isolate high-volume work without losing verifiable facts.

1API Message Structure

Вызов модели не сохраняет состояние сам по себе: harness каждый раз заново передает упорядоченный список сообщений с ролями system, user, assistant и tool, а определения инструментов отправляет отдельным полем. Статический префикс задает правила и доступные действия, а динамическая траектория накапливает запросы, решения модели, вызовы и результаты инструментов. Автор разбирает запрос о времени и погоде в Ванкувере: модель сначала возвращает два независимых tool call, harness выполняет их параллельно, связывает результаты через tool_call_id и повторно вызывает модель со всей историей. Главная работа агентного фреймворка в этом примере — корректно поддерживать messages, а не имитировать память внутри модели. Современная релевантность: В production-системе полезно хранить траекторию как типизированный журнал событий: не смешивать результаты инструментов с пользовательским текстом, сохранять исходные tool call и tool result, ограничивать число итераций и валидировать аргументы перед исполнением. Такой журнал упрощает воспроизведение сессий, аудит и точечное сжатие.A model call is stateless by itself: on every request, the harness resends an ordered list of messages with system, user, assistant, and tool roles, while tool definitions are supplied in a separate field. The static prefix establishes rules and available actions; the dynamic trajectory accumulates requests, model decisions, tool calls, and tool results. The author walks through a request for the time and weather in Vancouver: the model first returns two independent tool calls, the harness executes them in parallel, associates the results through tool_call_id, and calls the model again with the full history. In this example, the agent framework’s primary job is to maintain messages correctly, not to simulate memory inside the model. Modern relevance: In production, it is useful to store the trajectory as a typed event log: keep tool results separate from user text, preserve the original tool calls and results, cap the number of iterations, and validate arguments before execution. This kind of log makes sessions easier to replay and audit and supports targeted compression.

Модель похожа на специалиста без долговременной памяти: перед каждым решением ей заново выдают дело с правилами, перепиской и результатами проверок.The model is like a specialist with no long-term memory: before each decision, it must be given the entire case file again, including the rules, correspondence, and verification results.

each call is stateless; all information the model needs must be fully provided

2KV Cache-Friendly Context Design

KV Cache внутри инференса и Prompt Cache между API-запросами работают на разных уровнях, но требуют одного архитектурного свойства: неизменного токенового префикса. Поэтому системный промпт, определения инструментов, их порядок и chat template должны быть стабильны. Время можно получать инструментом или добавлять в хвост, состояние задачи и среды — через Agent Status Bar, а часто меняющуюся пользовательскую конфигурацию — через специализированный слой состояния. В книге приведен кейс клиентского агента на 100 000 диалогов в день: добавление Current time: {{now}} в системный промпт делало каждый префикс новым, после чего TTFT вырос с примерно 0,5 секунды до 3–5 секунд, а счет почти удвоился. Современная релевантность: Архитектуру контекста стоит делить на byte-stable и cache-breaking зоны ещё на этапе проектирования.The KV cache within inference and the prompt cache across API requests operate at different levels, but both depend on the same architectural property: an unchanged token prefix. The system prompt, tool definitions, tool order, and chat template should therefore remain stable. The current time can be obtained through a tool or appended near the end; task and environment state can be exposed through an Agent Status Bar; frequently changing user configuration can live in a dedicated state layer. The book describes a customer-service agent handling 100,000 conversations per day: adding Current time: {{now}} to the system prompt made every prefix unique, increasing TTFT from about 0.5 seconds to 3–5 seconds and nearly doubling the bill. Modern relevance: The context architecture should be divided into byte-stable and cache-breaking regions from the outset.

Кеш — это производственная линия: если изменить первую операцию, все последующие заготовки приходится пересобрать.A cache is like a production line: change the first operation, and every workpiece downstream has to be rebuilt.

once the system prompt is set, don’t change it

3Prompt Engineering and Prompt Injection

Хороший системный промпт организует поведение как исполнимый процесс: структурирует инструкции, задает приоритеты, явно определяет бизнес-границы и использует few-shot только там, где правило трудно формализовать. Но внешний текст нельзя считать инструкцией: веб-страницы, письма, документы и metadata образуют недоверенный слой данных и могут содержать prompt injection. В примере биллинга расплывчатая команда «выбери подходящий тип оплаты» заменяется на операционное правило, отделяющее снижение существующего счета от возврата и отмены подписки. Эксперимент с разрушением структуры промпта показывает падение успешности более чем на 30%, а удаление описаний tools — рост ошибок вызова на 45%. Для injection автор предлагает source tagging, строгие роли и sanitization только как первый слой защиты. Современная релевантность: Политики следует оформлять как проверяемые decision tables или SOP, а изменения оценивать абляциями.A good system prompt organizes behavior as an executable process: it structures instructions, establishes priorities, explicitly defines business boundaries, and uses few-shot examples only when a rule is difficult to formalize. External text, however, must not be treated as instruction: web pages, emails, documents, and metadata form an untrusted data layer and may contain prompt injections. In the billing example, the vague instruction “choose the appropriate payment type” is replaced with an operational rule that distinguishes reducing an existing bill from issuing a refund or canceling a subscription. An experiment that disrupted the prompt’s structure showed a drop in success rate of more than 30%; removing tool descriptions increased tool-call errors by 45%. For injection defense, the author presents source tagging, strict roles, and sanitization as only the first layer of protection. Modern relevance: Policies should be expressed as verifiable decision tables or SOPs, and changes should be evaluated through ablations.

3Dense–Sparse Hybrid Retrieval with Neural Reranking

Dense retrieval обеспечивает семантическое совпадение, sparse retrieval — точное лексическое совпадение, а production-grade RAG объединяет оба канала и переранжирует общий пул cross-encoder-моделью. Качество определяется не красотой финального ответа, а измеряемой вероятностью найти нужное свидетельство и поднять его вверх выдачи. До поиска документы режутся на chunks: слишком маленькие теряют субъект и контекст, слишком большие смешивают темы и размывают embedding. Dense bi-encoder сравнивает независимо вычисленные векторы; BM25 использует редкость термов, насыщение частоты и нормализацию длины. Поскольку cosine similarity и BM25 имеют разные шкалы, результаты сливаются нормализованной взвешенной суммой либо Reciprocal Rank Fusion. Современная релевантность: Начинать следует со structure-aware chunking, dense+BM25, RRF и reranker, а затем на собственном наборе запросов подбирать chunk size, overlap, top-k и бюджет reranking. Технические коды, имена и идентификаторы должны тестироваться отдельно от семантических и многоязычных запросов; мониторинг обязан различать retrieval miss и ошибку генерации.Dense retrieval provides semantic matching, sparse retrieval provides exact lexical matching, and production-grade RAG combines both channels before reranking the shared candidate pool with a cross-encoder model. Quality is determined not by how polished the final answer looks, but by the measurable probability of finding the required evidence and moving it toward the top of the results. Before retrieval, documents are split into chunks: chunks that are too small lose their subject and context, while chunks that are too large mix topics and dilute the embedding. A dense bi-encoder compares independently computed vectors; BM25 uses term rarity, frequency saturation, and length normalization. Because cosine similarity and BM25 operate on different scales, their results are merged through either a normalized weighted sum or Reciprocal Rank Fusion. Modern relevance: Start with structure-aware chunking, dense retrieval plus BM25, RRF, and a reranker, then tune chunk size, overlap, top-k, and the reranking budget on your own query set. Technical codes, names, and identifiers should be tested separately from semantic and multilingual queries; monitoring must distinguish a retrieval miss from a generation error.

4Structured Indexing and the Filesystem Paradigm

6Knowledge Discovery from Structured Datasets

Если экспертные знания распределены по тысячам случаев, одного document retrieval недостаточно: LLM сначала структурирует факторы, а статистический анализ обнаруживает прототипы, закономерности и относительную важность признаков. Современная релевантность: По аналогии этот pipeline можно проверять в поддержке, управлении инцидентами, медицине, страховании и контроле качества.When expert knowledge is distributed across thousands of cases, document retrieval alone is insufficient: an LLM first structures the factors, and statistical analysis then discovers prototypes, patterns, and the relative importance of features. 3Event-Driven Asynchronous Agent

1Pre-Training → Supervised Fine-Tuning → Reinforcement Learning

Возможности модели формируются последовательно. Pre-training минимизирует next-token loss на сыром корпусе и создаёт языковые, фактологические и reasoning priors. SFT использует ту же цель, но на демонстрациях input→ideal response и с loss masking вопроса, закрепляя формат, стиль и протокол. RL заменяет стандартный ответ функцией награды и повышает вероятность действий, которые приводят к хорошему результату. SFT обычно предшествует RL, потому что reward нельзя надёжно вычислить, если модель не выдаёт распознаваемый JSON или tool call. Это условная, а не универсальная последовательность: DeepSeek-R1-Zero показал direct RL на сильной базе, но получил poor readability и смешение языков; R1 вернул cold-start SFT. Современная релевантность: Практическая экстраполяция: перед обучением определить, какую способность нужно записать в веса. Pre-training minimizes next-token loss on a raw corpus and establishes linguistic, factual, and reasoning priors. SFT uses the same objective on input-to-ideal-response demonstrations, with the loss masked on question tokens, to instill format, style, and protocol. RL replaces the fixed target response with a reward function and increases the probability of actions that lead to good outcomes. SFT usually precedes RL because reward cannot be computed reliably if the model does not produce recognizable JSON or tool calls. This sequence is conditional rather than universal: DeepSeek-R1-Zero demonstrated direct RL on a strong base model, but suffered from poor readability and language mixing; R1 restored cold-start SFT. Modern relevance: A practical extrapolation is to decide before training which capability must be written into the weights. First test format, style, and a stable tool protocol with a small SFT/LoRA run; leave factual knowledge to RAG or continued pre-training; start RL only after reaching a measurable parseability threshold and establishing a working verifier.

3RLHF, RLVR, DPO, PPO, and GRPO

PPO и GRPO могут работать с обоими источниками: PPO получает тонкое advantage estimation через value network, GRPO дешевле оценивает rollout относительно группы. Современная релевантность: Практическая экстраполяция: сначала выбрать проверку цели — code/tests для RLVR, preference data для DPO/RLHF, generative evaluator для открытых задач — и лишь затем PPO или GRPO.The reward source and the optimization method are independent decisions. RLHF trains a Reward Model on pairs of human preferences and then optimizes the policy; RLVR uses a deterministic verifier. PPO and GRPO can work with either source: PPO obtains fine-grained advantage estimates through a value network, while GRPO estimates each rollout relative to its group at lower cost. DPO performs offline preference optimization without a separate Reward Model or online exploration. 4Data and Environment Matter More Than Algorithms

Политика учится не абстрактному миру, а конкретной распределённости задач, ответам симулятора и reward signal. 5Multi-Turn Credit Assignment, Process Rewards, and RLVP

В длинной ReAct-траектории итоговая награда запаздывает и не указывает, какое действие помогло или навредило. RLVP сохраняет outcome основной целью и добавляет детерминированные penalties или reachable partial credits конкретным действиям. Современная релевантность: Практическая экстраполяция: для платежей, кодовых изменений и внешних сообщений определить machine-verifiable path invariants, штрафовать только конкретные действия в состоянии перед ними и сохранять outcome как главную цель.In a long ReAct trajectory, the final reward arrives late and does not reveal which action helped or caused harm. 1Cascaded, End-to-End Omnimodal, and Full-Duplex Voice Architectures

Автор задаёт три парадигмы. Full-Duplex / Interactive Model отказывается от самой модели очередности: входной и выходной аудиопотоки обрабатываются одновременно, поэтому наложение речи и перебивание не требуют отдельной цепочки barge-in. 5Planning–Control Hierarchy and Action Chunking

Робот принимает решения в двух масштабах времени. Медленный Long-Horizon Planner декомпозирует цель и выбирает подцели; быстрый Vision-Language-Action Controller непрерывно преобразует изображение и инструкцию в управляющие сигналы. Это разделение отвечает на вопросы «что делать» и «как сделать» и не совпадает с MPS-разделением мышления и речевой артикуляции. Action Chunking компенсирует то, что VLA-инференс работает примерно на 1–10 Гц, тогда как традиционному управлению требуется 50–1000 Гц. Вместо одного действия модель генерирует короткую будущую траекторию; управляющий поток проигрывает её на высокой частоте, пока в фоне вычисляется следующая. OpenVLA в исходной одношаговой схеме работает примерно на 6 Гц. Экстраполяция (не утверждение книги): chunk следует делать прерываемым по watchdog-сигналам — появлению человека, расхождению ожидаемой и наблюдаемой позы, росту усилия или падению уверенности. Длина пакета может динамически уменьшаться у препятствий и увеличиваться на предсказуемом свободном участке.A robot makes decisions at two time scales. A slow Long-Horizon Planner decomposes the goal and selects subgoals; a fast Vision-Language-Action Controller continuously transforms images and instructions into control signals. This separation answers “what to do” and “how to do it” and is distinct from MPS’s separation of reasoning and speech articulation. Action Chunking compensates for VLA inference running at roughly 1–10 Hz while conventional control requires 50–1,000 Hz. Instead of producing one action, the model generates a short future trajectory; the control thread replays it at high frequency while the next chunk is computed in the background. The original single-step OpenVLA setup runs at about 6 Hz. Extrapolation (not a claim made by the book): A chunk should be interruptible through watchdog signals such as a person entering the scene, divergence between expected and observed pose, rising force, or falling confidence. Chunk length can shrink dynamically near obstacles and grow along predictable open segments.

6Cross-Embodiment Generalization and Sim2Real Transfer

VLA-контроллеры учатся главным образом имитацией человеческих демонстраций. Open X-Embodiment объединяет демонстрации более чем с 20 платформ, чтобы выучить общие паттерны захвата и размещения, после чего модель адаптируется небольшим набором данных конкретного робота. Sim2Real Transfer решает другой разрыв: симуляция неточно воспроизводит физику, изображение и динамику приводов. Domain Randomization варьирует трение, массу, освещение, текстуры, положение камеры, задержку и шум, чтобы реальный мир оказался внутри обучающего распределения. RT-2 and OpenVLA represent continuous actions as discrete action tokens and generate them autoregressively; π₀ uses flow matching to produce a continuous trajectory. 1Shared Context vs. Context Isolation

Главная архитектурная развилка — наследует ли новый агент полную траекторию предыдущего. Shared context не отбрасывает записи при передаче между ролями и упрощает короткие последовательные переходы, но быстро раздувает окно и переносит инерцию рассуждения; длинный контекст всё равно может вытеснить детали из внимания модели. Non-shared context требует явных интерфейсов, зато даёт модульность, изоляцию сбоев, параллельность и чистое внимание каждой роли. При shared context system prompt и tool set меняются, поэтому identity агента новая, хотя история едина. Книжный coding workflow переключает Requirements Analyst → Software Engineer → Code Reviewer через специальные transition tools, сохраняя детали требований и допускает rollback. Экстраполяция анализа: переключать режим по измеренным context occupancy, retrieval misses и handoff-loss rate, а не только по фиксированному порогу.The central architectural decision is whether a new agent inherits the previous agent's full trajectory. Shared context discards no records during role transitions and simplifies short sequential handoffs, but it rapidly expands the context window and carries reasoning inertia forward; even a long context can still push details out of the model's attention. Non-shared context requires explicit interfaces, but provides modularity, failure isolation, parallelism, and focused attention for each role. With shared context, the system prompt and tool set change, so the agent's identity is new even though the history remains unified. The book's coding workflow transitions from Requirements Analyst to Software Engineer to Code Reviewer through dedicated transition tools, preserving requirement details and allowing rollback. Analytical extrapolation: Switch modes based on measured context occupancy, retrieval misses, and handoff-loss rate rather than a fixed threshold alone.

2Peer, Manager, and Decentralized Collaboration Topologies

Топология определяет, где находится контроль и как течёт информация. Peer loop даёт взаимную проверку небольшой пары; Manager централизует декомпозицию, планирование, мониторинг и синтез; decentralized handoff распределяет решения о маршрутизации между специализированными агентами. Без общего контекста связь должна быть оформлена как контракт, а не как пересказ всей траектории. Proposer–Reviewer подходит итерационным артефактам и передаёт структурированную обратную связь. Manager обращается с подагентами как с инструментами: передаёт задачу и релевантный контекст, получает вывод, находки, пути к артефактам и проблемы, а полные трассы оставляет в локальных журналах. В planner–executor задачах, где ошибку декомпозиции трудно компенсировать исполнителю, глава рекомендует приоритизировать сильнейшую модель и лучший prompt для Planner/Manager. Экстраполяция анализа: описывать топологию машинно-читаемым графом с ownership, разрешёнными переходами и условиями завершения.Topology determines where control resides and how information flows. A peer loop enables mutual review by a small pair; a Manager centralizes decomposition, planning, monitoring, and synthesis; decentralized handoff distributes routing decisions among specialized agents. Without shared context, communication must be expressed as a contract rather than a retelling of the entire trajectory. Proposer–Reviewer is suitable for iterative artifacts and passes structured feedback. A Manager treats sub-agents as tools: it sends a task and relevant context, receives conclusions, findings, artifact paths, and problems, and leaves full traces in local logs. 4Multi-Agent Data Plane, Control Plane, and A2A Interoperability

Изолированная multi-agent system требует двух инфраструктур, независимых от выбранной топологии. Data plane передаёт долговечные артефакты через виртуальную файловую систему с разными областями видимости, жизненным циклом и правами доступа. A2A стандартизирует обнаружение возможностей, жизненный цикл задач и обмен артефактами через границы доверия, но не заменяет внутренние параметры, файлы и шину сообщений. Virtual filesystem монтирует private scratchpad каждого агента, persistent shared workspace, permission-bound external resources и stable read-only system resources. A2A standardizes capability discovery, task lifecycles, and artifact exchange across trust boundaries, but does not replace internal parameters, files, or the message bus. 6Agent Society and Model–Harness Co-Evolution

При долгом свободном взаимодействии возникают социальные, экономические и стратегические эффекты, которых нет в локальной оркестрации задач.
