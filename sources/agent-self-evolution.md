# Самоэволюция агента без изменения весов

Глава 8 определяет self-evolution не как магическое «самообучение» модели, а как внешнее обучение: агент превращает опыт в постоянные, извлекаемые и исправляемые ресурсы. У механизма две оси — дистилляция знаний из успешных и неуспешных траекторий и расширение границы возможностей через обнаружение и создание инструментов. Это дополняет post-training, но не заменяет его.

## 1. Externalized Learning среди трёх парадигм

Длинный контекст сам по себе не учит. Attention хорошо извлекает конкретную запись, но не обязан сжимать сотни эпизодов в общий принцип; рост истории дополнительно вызывает context rot. Поэтому обучение требует отдельного вычислительного шага: завершить задачу, отрефлексировать опыт, сжать и структурировать его, сохранить, а при следующей задаче извлечь.

Книга различает три носителя. In-Context Learning быстро адаптирует поведение, но исчезает после сессии. Post-training записывает стабильные стратегии в параметры, даёт низкую задержку и высокую повторяемость, но дорого и медленно обновляется. Externalized Learning сохраняет знания в файлах, базах и инструментах: они постоянны, интерпретируемы и редактируемы. Для факта подходит knowledge-base entry, для частой параметризованной процедуры — dedicated code tool, для меняющегося процесса со стратегическим суждением — Skill. Прямое применение из книги: после задачи тратить compute на активную дистилляцию, а не архивировать сырую траекторию как «обучение». **Экстраполяция анализа:** каждому новому правилу задавать owner, provenance, срок пересмотра и метрику повторного использования.

## 2. Дистилляция успехов, неудач и процедурная память

Strategy Summary сохраняет не финальный ответ, а переносимый метод: использованные приёмы, ловушки и ключевую последовательность. В эксперименте GAIA успешная траектория рефлексируется, векторизуется и извлекается как аналогичный прецедент для новой задачи. Критерий записи — transferability: решение одного уникального input не должно засорять память.

Workflow Recording превращает повторяемую браузерную работу в параметризованный state machine. Перед каждым действием verification predicate сверяется с живым экраном; при расхождении управление возвращается полноценному агенту. После компиляции среда сбрасывается и workflow проигрывается с нуля до записи в библиотеку. Эта pre-storage verification не даёт сохранить программу, которая формально нажала все кнопки, но не выполнила цель. Повторное исполнение без model calls в описанном механизме ускоряется в 8,5–13 раз. Reflexion дополняет успехи failure lessons: отрицательные правила и распознанные error patterns возвращаются в следующий похожий запуск. **Экстраполяция анализа:** считать transfer success отдельно от task success и автоматически понижать доверие к процедуре после каждого fallback.

## 3. Skills и фоновая консолидация

Skill — переносимое, человекочитаемое руководство, которое загружается по требованию. Оно быстрее retraining, доступно аудиту и переживает смену модели. Skill Creator превращает создание Skills в метаспособность: выясняет сценарии, выделяет переиспользуемые ресурсы и собирает пакет из `SKILL.md`, scripts, references и assets. Claude Code-подобный project guide аналогично закрепляет архитектуру, команды тестирования и локальные соглашения репозитория.

Накопление и организация должны происходить в разные окна. По описанию автора, основанному на поведении публичной версии Claude Code и community analysis, а не официальной спецификации, Sleep Learning запускает отдельного фонового агента после достаточного числа новых сессий: Orient читает индекс, Gather ищет новые и противоречивые факты, Consolidate объединяет их и нормализует даты, Prune & Index удаляет устаревшие ссылки. Консолидация ограничена каталогом памяти, защищена locks и rollback. Hermes развивает эту модель до resident service с Prompt Memory, Session Retrieval, Skill Library и user modeling. Прямое применение из книги: для автоматически создаваемых Hermes Skills предпочитать incremental patches, а память консолидировать асинхронно через merge, prune и rollback. **Экстраполяция анализа:** хранить reversible diff каждой консолидации и давать пользователю просматривать спорные merge/delete.

## 4. System Prompt Learning

System prompt — самый прямой внешний носитель поведенческого опыта. Один scalar reward говорит лишь «верно/неверно», а языковой post-mortem объясняет причину: например, сначала проверить личность, а затем начинать возврат. Автор формулирует задачу как уточнение границ правил через edge cases. Если агент слишком часто переводит policy dispute человеку, Coding Agent читает текущий prompt, добавляет узкую положительную границу и отрицательное правило, проверяет непротиворечивость и отдаёт diff эксперту.

DSPy, OPRO и GEPA оптимизируют prompts офлайн на scored eval sets; case-by-case diff подходит для production-исключений с human review. Они дополняют друг друга: batch optimization создаёт baseline, diff-подход продолжает контролируемую эволюцию. Отдельная инженерная опора — persistent task state: Initializer сохраняет JSON feature list и progress, а последующие сессии завершают по одному пункту; готовность наступает только при `passes: true` у всех features. **Экстраполяция анализа:** любое изменение prompt должно проходить retained regression set и иметь простой rollback.

## 5. Proactive Tool Discovery

Полная инъекция тысяч schemas переполняет контекст и ухудшает выбор. Retrieval по исходному запросу тоже ограничен: потребность в новом инструменте может проявиться лишь посередине задачи. MCP-Zero предлагает active discovery: агент формулирует capability gap естественным языком, поиск сначала выбирает server, затем tool, и только после этого схема попадает в контекст. В приводимом результате около 2800 инструментов дают примерно 98% экономии токенов против полной инъекции.

Skills предлагают более лёгкий путь: на старте видны только name и description, затем агент читает нужный файл и его ссылки обычными grep/read. Переменную часть следует добавлять в конец истории, сохраняя стабильный KV-cache prefix. При низкой уверенности discovery должен вернуть `not found`, после чего агент переформулирует запрос, импровизирует базовыми средствами или создаёт инструмент. **Экстраполяция анализа:** измерять false discovery, время до обнаружения gap и долю успешного reuse, а не только финальную точность.

## 6. От пользователя инструментов к их создателю

Принцип Alita — Minimal Predefinition, Maximal Self-Evolution. Конкретный прототип Experiment 8-5 начинает с `web_search`, `read_webpage`, `code_interpreter`, `create_tool` и `search_tools`, а остальное агент находит или пишет. В книжном примере агент обнаруживает `youtube-transcript-api`, читает README, тестирует библиотеку, оборачивает её в MCP-tool, извлекает subtitles и сохраняет интеграцию. Voyager показывает цикл в Minecraft: curriculum generator выбирает достижимую следующую цель, iterative prompting исправляет код, а иерархическая skill library ускоряет дальнейшее исследование; сообщается о 3,3-кратном росте числа уникальных предметов против baselines.

Рост идёт на трёх уровнях: tools, knowledge о библиотеках и APIs, strategy выбора и тестирования. Но повторное использование умножает и ошибку. Книга выделяет supply-chain attack, capability drift, quality degradation и persistent memory poisoning. Защита требует sandbox, security scan, allowlist, pre-write review, source tags, разделения reference data и instructions, проверки при retrieval и возможности удалить запись. **Экстраполяция анализа:** новый инструмент не должен получать network/secret access до изолированных тестов, provenance review и явного promotion gate.
