# Инструменты и код как метаспособность агента

Главы 4–5 показывают единый переход: сначала модель получает надёжный интерфейс к миру, затем учится создавать новые интерфейсы сама. Инструмент превращает вероятностное намерение LLM в проверяемое действие; код делает этот механизм расширяемым. Поэтому качество агента определяется не только интеллектом модели, но и тем, какие операции ей доступны, как устроена обратная связь и может ли система безопасно пережить ошибку или внешнее событие.

## 1. ACI: интерфейс между агентом и компьютером

Agent–Computer Interface — аналог HCI для взаимодействия агента с компьютером: инструменты проектируются под цели и ограничения агента, а не как механические обёртки над API. Автор предлагает выбирать форму способности по сложности параметров, частоте изменений и силе модели: стабильную сложную операцию оформить отдельным инструментом, а простую или быстро меняющуюся — навыком и универсальным исполнителем. Близкие операции с одинаковой формой полезно объединять: PDF, DOCX и PPTX можно читать через `read_document`; OCR и видеоанализ лучше разделить из-за разных параметров и задержек.

Обобщение имеет предел. `code_interpreter` заменяет десятки калькуляторов, но запись в production-базу требует узкого инструмента с правами и аудитом. Описание должно сообщать не только функцию, но и момент применения, границы, отрицательные примеры, формат результата и задержку. Критическое правило fidelity запрещает скрыто менять аргументы: если модель передала строку или команду, инструмент не должен незаметно подставлять дополнительные параметры. Прямое применение из книги: pagination, явное усечение длинного вывода с сохранением полного результата в файл, идемпотентные ключи и execute–validate–feedback. **Экстраполяция анализа:** версионировать JSON Schema инструмента и тестировать контракт на реальных траекториях до выката.

## 2. MCP, Skills и обнаружение по требованию

Model Context Protocol стандартизирует соединение клиента и серверов инструментов, ресурсов и prompts через локальный stdio или удалённый Streamable HTTP. Его ценность — переносимость интеграции, но не планирование жизненного цикла задачи. Если загрузить определения всех инструментов сразу, они съедают контекст и ухудшают выбор. Поэтому книга разводит роли: MCP решает совместимость, Skills уменьшают перегрузку выбора, а dynamic tool discovery извлекает подробное определение только по необходимости. В приведённом эксперименте такой поиск поднял точность Opus 4 на tool-use benchmark с 49% до 74%.

Каждый сторонний сервер создаёт новую границу доверия: возможны description poisoning, tool shadowing и неправомерное использование OAuth-токенов или API-ключей. Прямое применение из книги: проверять сервер, фиксировать версии, выдавать минимальные права и ставить структурированный Sidecar-классификатор перед исполнением опасных вызовов. **Экстраполяция анализа:** вести реестр происхождения инструментов и запрещать неизвестному серверу перекрывать уже доверенное имя.

## 3. Событийный асинхронный агент

Реальный мир не соблюдает синхронную схему «tool call — немедленный tool result». Звонок может запросить код подтверждения, пользователь — прервать долгий поиск, а webhook — сообщить об аварии. В примере PineClaw канал событий сокращает задержку реакции с минут до секунд: мир активно уведомляет агента, а не ждёт очередного Heartbeat.

Архитектура унифицирует пользовательские сообщения, результаты инструментов, callbacks и таймеры в структурированный поток. Срочное событие отменяет текущую ветку, рутинное становится в очередь, независимое дешёвое задание запускается параллельно. Долгая операция должна вернуть `task_id`, а завершение прийти отдельным событием; нужны status и cancel. Прямое применение из книги: placeholder сохраняет синтаксис нынешнего API при прерывании, но это временный компромисс и модель может принять заглушку за настоящий результат. **Экстраполяция анализа:** хранить durable event log и correlation ID, чтобы после рестарта точно восстановить причинную цепочку.

## 4. Coding Agent и его Harness

Для открытых задач книга сводит основу к Coding Agent плюс файловая система. Минимальный набор — code interpreter, Bash, read, write, edit, glob и grep. Файлы служат одновременно рабочей памятью, артефактами и проверяемым состоянием; процесс можно перезапустить, если состояние воспроизводится из репозитория, инструкций и промежуточных результатов.

Однако способность писать код ещё не создаёт надёжного агента. Harness задаёт baseline приёмки, границы исполнения, быстрые сигналы обратной связи и rollback. Рабочий цикл: прочитать проектные инструкции, уточнить требование, для сложной задачи согласовать дизайн, реализовать, запустить тесты, исправить ошибки, выполнить self-review и синхронизировать документацию. Линтер, типы, тесты, Git и структурированные tool errors превращают ошибку в следующий наблюдаемый шаг. Повторяющиеся одинаковые вызовы обнаруживаются по fingerprint, retry ограничивается, а circuit breaker завершает death spiral. Прямое применение из книги: ограничения, которые можно выразить кодом, должны исполняться, а не оставаться советом. **Экстраполяция анализа:** публиковать метрики recovery rate, повторов и аварийных остановок рядом с обычным task success.

## 5. Код как формальное мышление и исполняемая политика

LLM хорошо понимает естественный язык, но вероятностное рассуждение ненадёжно в арифметике и строгой логике. Разделение труда делает модель переводчиком задачи в Python, SymPy или constraint solver, а интерпретатор — источником точного проверяемого результата. В примере с 40 учениками текстовое решение вычитает пересечение не из того множества и получает 14, тогда как код вычисляет 8.

Тот же принцип переносится на бизнес-правила. Политика отмены авиабилета существует в трёх слоях: естественный язык помогает объяснять; параметры инструмента заставляют пройти checklist; серверная проверка по данным БД и серверному времени блокирует необратимую ошибку. Самоотчёт модели `expected_*` полезен для аудита, но не считается истиной. Прямое применение из книги: codified gatekeeper перед отменой, переводом денег или удалением. **Экстраполяция анализа:** прогонять правила на исторических кейсах и формально контролировать полноту ветвей до изменения политики.

## 6. Код как универсальный адаптер и механизм порождения

Метаспособность создаёт новые способности на лету. Агент генерирует Slidev-презентацию или Blender/FFmpeg-сценарий, после чего независимый Reviewer смотрит рендер и возвращает структурированные исправления. Для нестабильного API код становится «универсальным клеем»: строит HTTP-клиент, преобразует схему и адаптирует новый формат логов. В Artifact pattern SQL идёт из агента системному исполнителю, а данные — из БД в интерфейс, минуя пересказ LLM. Безопасность generated SQL должна обеспечиваться отдельно: read-only правами, allowlist операций, валидацией и при необходимости sandbox или approval.

Иллюстрация Figure 5-8 сравнивает десять уточняющих раундов с одной отправкой динамической формы; это не контролируемый эксперимент. Для стандартных интерфейсов A2UI-подобный декларативный manifest безопаснее произвольного HTML: клиент разрешает только доверенные компоненты. Крайняя точка — bootstrapping: агент копирует проверенный каркас другого агента, меняет prompt, tools и бизнес-логику, затем тестирует результат. Прямое применение из книги: модификация эталона наследует хорошие архитектурные решения лучше генерации с нуля. **Экстраполяция анализа:** подпись шаблона, изолированная сборка и обязательный eval-gate должны предшествовать запуску порождённого агента с реальными правами.
